주머니 속 한국어 전자책 리더를 직접 만들었다

2026.07.18AI 90%방향과 승인 판단은 사람, 조사·구현·검증·초안 작성은 AI와 협업.

예전에 Xteink X4에 한글 책을 넣어 읽는 과정을 정리했다. 그때의 목표는 일단 한글 EPUB을 여는 것이었다. 이번에는 한 걸음 더 갔다. 한국어로 오래 읽기 편하고, 책을 덮으면 내가 읽고 있는 책이 자동으로 kimtoma.com에 표시되는 개인용 펌웨어를 만들었다.

이름은 CrossMux KR. 아직 운영용 최종판은 아니다. 실제 기기에 설치하고 서버 인증까지 성공했지만, 내가 정한 메모리 안전 기준을 348바이트 넘지 못해 마지막 릴리스를 멈춘 상태다. 동작하는 것과 안심하고 매일 쓰는 것은 다른 문제라서, 그 작은 차이를 그냥 넘기지 않기로 했다.

한글이 보이는 것과 한국어로 읽기 좋은 것은 다르다

Xteink X4는 4.3인치 이잉크 화면과 물리 버튼, ESP32-C3를 사용하는 아주 작은 전자책 리더다. 안드로이드 기기처럼 앱을 설치하는 방식이 아니라, 제한된 플래시와 약 380KB의 RAM 안에서 펌웨어가 EPUB 파싱부터 화면 그리기, Wi-Fi, HTTPS까지 모두 처리한다.

한글 글리프 몇 개를 추가하면 제목과 본문이 보이기 시작한다. 하지만 실제 한국어 책을 오래 읽으려면 다른 문제가 나온다.

  • 목록에서 작은 글꼴을 쓸 때 제목의 일부가 네모로 보인다.
  • EPUB 원문에 있던 띄어쓰기가 CJK 조판 과정에서 사라질 수 있다.
  • 한글 11,172자를 모든 크기에 넣으면 펌웨어 용량이 빠르게 커진다.
  • 네트워크 동기화를 추가하면 TLS가 읽기 엔진과 같은 작은 힙을 사용한다.

그래서 이번 프로젝트의 목표를 “한글 지원”이 아니라 한국어 장문 독서에 맞춘 제품 구성으로 다시 정의했다.

세 개의 오픈소스 저장소에서 출발했다

이 펌웨어의 계보는 세 저장소로 이어진다.

  1. crosspoint-reader/crosspoint-reader — Xteink용 오픈소스 리더의 원류
  2. 0x1abin/crossmux — CrossPoint를 확장한 포크
  3. ryokun6/crossmux — CJK, 세로쓰기, 한국어 빌드, 그레이스케일 렌더링을 발전시킨 포크

처음에는 포크 하나를 그대로 선택하면 될 것 같았다. 코드를 비교해보니 답은 조금 달랐다. 원류의 안정적인 하드웨어·복구 구조를 기준으로 삼고, ryokun6/crossmux의 한국어 빌드인 gh_release_ko를 출발점으로 쓰되, 내가 사용하지 않는 서비스는 한국어 SKU에서 덜어내는 편이 맞았다.

게임을 지우면 용량이 크게 줄 것 같았지만, 확인해보니 게임 소스는 이미 모든 빌드에서 제외되어 있었다. 실제로 줄일 수 있었던 것은 AirPage/MQTT, ryOS 전용 클라우드 동기화와 설정, 기본으로 추가되는 ryOS Books 카탈로그였다. 일반 OPDS, 파일 전송, 저장된 Wi-Fi, SD 카드 글꼴, 로컬 독서 통계와 OTA 복구는 남겼다.

이번 펌웨어에서 지키기로 한 것

한국어 글꼴은 크기별 역할을 나눴다.

크기 수록 범위 용도
8·10·12pt KS X 1001 공통 한글 2,350자 + UI 문자 제목, 저자, 목록, 작은 본문
14pt 현대 한글 11,172자 + 교육용 기초 한자 1,800자 기본 본문
16·18pt UI 문자와 현대 자모 큰 UI

모든 크기에 모든 글자를 넣는 대신, 목록에서 자주 만나는 한국어를 작은 글꼴에 보강하고 기본 본문 크기에 전체 현대 한글을 유지했다. 내장 글꼴은 Resource Han Rounded KR을 사용한다.

띄어쓰기는 EPUB 파서 쪽을 고쳤다. 중국어처럼 글자 사이 공백이 없어도 자연스러운 CJK와, 공백이 단어 경계인 한국어를 같은 규칙으로 처리하면 안 된다. 소스에 실제 공백이 있으면 보존하고, 한국어 section cache 버전도 올려 이전 파싱 결과가 남지 않게 했다.

기능을 더하는 기준도 정했다.

네트워크가 실패해도 책은 열려야 하고, 로컬 독서 기록은 먼저 저장되어야 한다.

이 원칙이 이후의 자동 연동 구조를 거의 결정했다.

책을 덮으면 웹사이트의 READING이 바뀐다

kimtoma.com 홈에는 내가 읽고 있는 책을 표시하는 READING 영역이 있다. 이전에는 마크다운과 코드에 직접 책 정보를 적었다. 책을 바꿀 때마다 수정하고 다시 빌드해야 했고, 홈의 목록 보기와 위젯 보기가 서로 다른 책을 보여주기도 했다.

CrossMux KR에서는 다음 중 하나를 만족한 독서 세션만 동기화한다.

  • 3분 이상 읽음
  • 진행률이 1%p 이상 증가
  • 책을 완료함

책을 잠깐 잘못 열었다 닫는 행동은 현재 책을 바꾸지 않는다. 같은 책과 같은 진행률도 다시 보내지 않는다.

책을 덮음
  → 기존 로컬 독서 통계를 먼저 저장
  → 최신 상태 한 건을 SD 카드 큐에 저장
  → 리더와 화면 메모리를 정리
  → 저장된 Wi-Fi로 HTTPS 요청
  → Cloudflare Worker가 D1에 현재 책을 기록
  → kimtoma.com의 목록과 위젯이 같은 상태를 표시

기기는 현재 상태를 api.kimtoma.com으로 보낸다. Cloudflare Worker는 기기 전용 토큰의 hash를 확인하고, D1에는 device·book·event·current를 분리해 저장한다. EPUB 원본 표지가 새로 필요할 때만 JPG나 PNG를 R2에 올린다.

각 요청에는 SD 카드에 보존되는 sequence가 있다. 같은 요청이 다시 오면 duplicate, 오래된 번호면 stale로 처리한다. 전원이 꺼지거나 응답을 받기 전에 연결이 끊겨도 현재 책이 과거 상태로 되돌아가지 않게 하기 위해서다.

사이트는 다음 순서로 데이터를 고른다.

  1. 공개 current API
  2. 브라우저가 마지막으로 성공한 값
  3. 빌드에 포함된 공통 기본값

API가 실패해도 빈칸이나 오류 배너 대신 마지막으로 알던 책이 그대로 남는다. 목록 보기와 위젯 보기는 하나의 ReadingProvider를 공유해서 서로 다른 책을 표시하지 않는다.

380KB 안에서 HTTPS를 실행하는 일

가장 어려운 부분은 API가 아니라 메모리였다. EPUB 파서, 글꼴 캐시, 표지, 홈 화면과 TLS가 모두 ESP32-C3의 작은 힙을 나눠 쓴다. 데스크톱이나 스마트폰에서는 눈에 띄지 않을 크기의 버퍼가 여기서는 부팅과 재시작을 가르는 차이가 된다.

동기화 worker가 책을 읽는 객체를 계속 붙잡지 않도록 수명주기를 바꿨다.

  1. 로컬 독서 통계를 저장한다.
  2. 제목·저자·진행률 같은 작은 값만 큐에 복사한다.
  3. EPUB과 렌더러가 소유한 메모리를 해제한다.
  4. 홈 화면이 안정된 뒤 Wi-Fi와 HTTPS를 시작한다.
  5. 성공과 실패에 관계없이 HTTP와 Wi-Fi 자원을 정리한다.

표지도 RAM에 통째로 올리지 않는다. EPUB 안의 원본 JPG/PNG를 1KB 단위로 SD 카드에 스트리밍하고, 서버가 필요하다고 답할 때만 전송한다.

기본 TLS 설정만 바꾼다고 실제 펌웨어에 반영되는 것도 아니었다. ESP32 도구 체인이 이미 컴파일한 mbedTLS archive를 링크하고 있었기 때문이다. 빌드에 사용되는 실제 archive를 찾아 수신 버퍼 16KiB, 송신 버퍼 4KiB의 비대칭 설정으로 다시 만들고, 최종 map과 disassembly에서 그 값이 들어갔는지 확인했다.

실제 Xteink4에 설치했다

최종 진단용 firmware.bin은 약 5.97MB였다. Xteink4의 OTA 슬롯 하드 상한 안에 들어왔고, 한국어 글꼴·독서 동기화·설정 UI·TLS 진단 로그를 함께 포함했다.

CrossPoint web flasher에서 Xteink X4와 custom firmware를 선택했다. USB 시리얼 장치를 연결하자 partition table 검증, OTA 정보 읽기, firmware 쓰기, boot partition 갱신, 연결 해제가 차례로 끝났다.

재부팅 후 기기는 저장된 Wi-Fi에 연결했고, 인증 전용 요청에 HTTP 200을 받았다. 요청 뒤에는 네트워크 수명주기 정리 로그도 확인했다. 서버의 현재 책과 sequence는 바뀌지 않았다. 인증 검사는 연결만 확인하고 실제 독서 상태를 갱신하지 않도록 설계했기 때문이다.

여기까지만 보면 성공이었다.

348바이트 때문에 멈췄다

이번 릴리스의 최소 여유 힙 기준은 50KiB, 즉 51,200바이트였다. 실제 기기에서 TLS 요청 중 측정한 최저값은 50,852바이트. 348바이트가 부족했다.

요청이 끝난 뒤 여유 힙은 137,872바이트로 회복됐고 API도 정상 응답했다. 지금 상태로도 한 번의 동기화는 동작한다. 하지만 최소값이 기준 아래라는 사실은 반복 사용, 더 큰 인증서, 다른 책의 메타데이터, 힙 단편화가 겹칠 때 여유가 거의 없다는 뜻이다.

기준을 49KiB로 낮추면 바로 “통과”라고 말할 수 있다. 그러지 않기로 했다. 이 기준은 결과를 본 뒤 만든 숫자가 아니라 구현 전에 정한 릴리스 게이트다. 작지만 예측하기 어려운 기기일수록, 스스로 만든 기준을 지키는 편이 낫다.

그래서 현재 Xteink4에는 진단 펌웨어가 설치되어 있고, 운영용 펌웨어 확정은 보류했다.

다음 버전에서 할 일

남은 작업은 구조를 다시 만드는 일이 아니다.

  1. 1~2KiB의 추가 메모리 여유를 확보한다.
  2. 같은 조건에서 최소 힙이 50KiB를 넘는지 다시 측정한다.
  3. 진단 로그를 제거한 운영용 이미지를 빌드하고 재설치한다.
  4. 실제 책을 3분 이상 읽거나 1%p 넘긴 뒤 닫아 sequence 증가를 확인한다.
  5. kimtoma.com의 목록과 책 위젯이 동시에 바뀌는지 확인한다.

이번 작업을 하면서 주머니 속의 작은 리더가 더 흥미로워졌다. 화면에 한글을 띄우는 기기에서, 읽기 경험과 데이터의 흐름을 내가 원하는 방식으로 설계할 수 있는 기기가 됐다.

아직 마지막 348바이트가 남았다. 다음 글은 그 숫자를 넘긴 운영용 펌웨어 이야기였으면 좋겠다.
