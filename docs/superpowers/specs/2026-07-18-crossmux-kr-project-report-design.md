## 1. 목표

CrossMux KR 펌웨어와 `kimtoma.com` 독서 현황 연동 프로젝트를 다음 세 가지 관점으로 기록하고, 각 문서를 Markdown과 독립 실행형 HTML로 제공한다.

문서는 아직 운영용 릴리스가 완료된 것처럼 서술하지 않는다. 2026-07-18 실기기 진단 펌웨어는 설치와 API 검증에 성공했지만, 최소 여유 힙이 50,852바이트로 승인 기준 51,200바이트보다 348바이트 부족하다. 이 값과 다음 최적화 필요성을 세 문서에 목적에 맞는 깊이로 반영한다.

3. 사용자 의사결정 타임라인
4. 펌웨어 구조: 한국어 조판·글꼴·기능 감량·로컬 우선 동기화
5. API·데이터 모델·표지·sequence·오류 처리

7. 실제 구현 산출물과 브랜치/작업 트리
8. 빌드·플래시·실기기 검증 결과
9. 성공, 미완료, 다음 단계의 명확한 구분
10. 관련 문서와 저장소 링크

사실과 계획을 섞지 않는다. 완료된 항목, 진단 단계까지 완료된 항목, 아직 운영 릴리스 전에 남은 항목을 별도 상태로 표시한다.

## 5. kimtoma.com 블로그 글

Markdown은 실제 사이트의 `content/posts/YYYY-MM-DD-slug.md` 규칙과 `gray-matter` 파서를 따른다.

```yaml
---
layout: post
title: "주머니 속 한국어 전자책 리더를 직접 만들었다"
ai: true
ai_pct: 90
ai_note: "방향과 승인 판단은 사람, 조사·구현·검증·초안 작성은 AI와 협업."
---
```

본문은 기존 `2026-02-18-xteink-x4-korean-ebook-guide.md`의 후속 글처럼 읽히는 1인칭 개발기로 작성한다. 구조는 다음과 같다.

1. 왜 기존 한글화만으로 부족했는가
2. CrossPoint Reader에서 CrossMux KR까지 이어진 계보
3. 이번 프로젝트에서 정한 제품 원칙
4. 한글 조판과 작은 글꼴의 현실
5. 책을 덮으면 사이트의 `READING` 상태가 바뀌는 구조
6. ESP32-C3의 메모리 제약과 TLS 최적화
7. 실제 Xteink4 플래시와 API 200 검증
8. 348바이트 부족 때문에 운영 릴리스를 멈춘 이유
9. 다음 단계와 오픈소스 링크

공개 글에서는 다음 정보를 제거하거나 일반화한다.

- 토큰과 Authorization 값
- 로컬 IP, 장치 MAC, USB 포트명
- 개인용 관리 명령과 로컬 절대 경로
- 서버 운영에 불필요한 내부 로그 전체

대신 재현에 필요한 공개 저장소, 웹 플래셔, 펌웨어 크기, 메모리 측정값, API 처리 원칙은 남긴다. HTML 미리보기는 kimtoma.com의 밝은/어두운 테마 색상, 1.35rem 본문 계층, 초록색 primary, 반응형 표·코드 블록을 반영한다. 실제 사이트 코드를 복제하지 않고 게시 결과를 검토하기 위한 독립 미리보기로 만든다.

## 6. 생성 방식

- 사람이 읽을 세 문서의 Markdown을 정본으로 둔다.
- 세션 원문은 작은 생성 스크립트가 JSONL에서 허용된 역할만 추출해 만들도록 해 누락과 수작업 변형을 피한다.
- HTML은 Markdown 정본을 변환한 뒤 문서 종류별 템플릿에 삽입한다.
- 블로그 Markdown은 `kimtoma.com`의 `gray-matter` 로더가 읽을 수 있는지 실제 로더 또는 동등한 파서로 확인한다.
- 게시나 배포는 수행하지 않는다. 최종 blog Markdown은 복사 즉시 `content/posts/`에 둘 수 있는 상태로 제공한다.

## 7. 검증 기준

### 내용

- 프로젝트 요약의 파일명, 빌드 크기, API 상태, 힙 측정값이 실제 결과와 일치한다.
- 완료와 미완료가 구분되며 운영 릴리스 완료를 과장하지 않는다.
- 대화 원문의 사용자/assistant 메시지 수와 추출 결과가 일치한다.
- 첫 사용자 요청과 마지막 승인 메시지가 모두 존재한다.
- 비공개 정보가 블로그 글에 포함되지 않는다.

### 형식

- Markdown 3개와 HTML 3개가 모두 존재한다.
- 모든 HTML에 `lang="ko"`, UTF-8, viewport, title이 있다.
- Markdown 내부 링크와 HTML 상대 링크가 깨지지 않는다.
- HTML에서 표, 코드, 인용문, 긴 대화가 화면 폭을 넘겨 본문을 깨뜨리지 않는다.
- 블로그 front matter가 `kimtoma.com/src/lib/posts.ts`에서 정상 파싱된다.

### 렌더링

- 세 HTML을 실제 브라우저에서 데스크톱과 좁은 화면으로 확인한다.
- 한글 깨짐, 잘린 표, 겹치는 요소, 코드 블록 overflow를 확인한다.
- 블로그 Markdown을 kimtoma.com 로컬 빌드에 임시 연결하거나 동등한 렌더러로 확인하되 게시·배포하지 않는다.
